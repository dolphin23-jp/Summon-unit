# AGENTS.md

このリポジトリでCodexまたは他の実装エージェントが作業する際の最優先規約です。

## 1. 仕様の優先順位

1. ユーザーから当該タスクで明示された指示
2. `tasks/Txxx_*.md` の対象タスク
3. `docs/` の正式仕様
4. 既存テスト
5. 既存実装

仕様と実装が矛盾する場合、勝手に仕様を変更せず、矛盾を報告してください。

## 2. 一度に一つのタスク

- 一つのタスクにつき一つのブランチ、コミット群、PRを基本とする
- 対象外の機能を先回りして実装しない
- 将来必要そうという理由だけで抽象化を増やしすぎない
- ただし、確定済みの拡張点を閉ざすデータ構造にはしない

## 3. 禁止事項

- UIコンポーネント内に戦闘ルールを書く
- モンスター表示名で条件分岐する
- 配列位置を永続IDとして保存する
- AI判断に無目的な乱数を入れる
- テストを削除・弱体化してCIを通す
- 同種編成制限や総編成コスト上限を追加する
- 開花技の解放を長時間周回へ変更する
- 高レアを低レアと強制的に同格化する
- 仕様外の大規模リファクタリングを混在させる
- 対象種別・分類フラグのような識別子を、将来値が増える前提のままリテラル型1値や固定enumに閉じない。拡張可能な文字列IDまたは判別可能なunion型で表現する。

## 4. アーキテクチャ境界

- `battle/`: 純粋な戦闘状態遷移。DOM・React・IndexedDBへ依存しない
- `content/`: モンスター、技、レシピ、ステージ等のマスターデータ
- `ai/`: 戦闘状態と行動候補を入力し、評価結果を返す
- `save/`: ゲーム状態の保存・移行。UIに依存しない
- `ui/`: 状態の表示とユーザー入力。ルールの正本にはしない
- `progression/`: プレイヤーの可変進行状態（所持個体・経済・研究進捗・地域進行）。content/のマスターデータ、save/の永続化とは分離する

## 5. 決定論

- 同じ初期状態、戦闘シード、手動入力から同じイベント列を生成する
- 同点解決は固定規則を使う
- ランダム技だけ明示的にシード付き乱数を使う
- すべての重要な状態変化をイベントログに残す

## 6. テスト

各タスクで最低限以下を行います。

- 追加ロジックの単体テスト
- 境界値テスト
- 決定論テスト（関係する場合）
- 既存テスト全件
- `pnpm lint`
- `pnpm build`

## 7. タスク完了報告

完了時は以下を報告します。

- 変更内容
- 変更ファイル
- 実行したテストと結果
- 手動確認方法
- 仕様上の判断
- 未完了・残課題
- 次に着手可能なタスク

未実装部分を完了したように表現してはいけません。
