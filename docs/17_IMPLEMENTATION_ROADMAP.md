# 17. Implementation Roadmap（v2）

本書は「1実装単位＝Codexへ渡す1タスク」の粒度で構成します。全33単位（T016～T048）。各単位内のチェックリストは、実装・コミット・レビューの内訳であり、単位をさらに分割する場合の境界線でもあります。

【単独】印の5単位は、壊れると後続全体へ波及する基盤です。束ねず小さく実装し、重点的にレビューします。

## 完成の定義

「完成」は `docs/14_VERTICAL_SLICE.md` の合格条件をすべて満たした垂直スライスのVercel公開（v0.1）です。探索遠征・R6以降・PvP・クラウド同期は対象外です（`docs/DECISIONS.md` 後回し項目）。

## 現在地

- Phase 0～1（T001～T015）完了
- 決定論的戦闘カーネル（盤面・タイムライン・単体技・移動・DIRECT遮蔽・最小AI・イベントログ）
- ヘッドレスデモと6×3再生UI、CI・Vercel稼働

## 技術スタック

初期標準（変更なし）:

- Vite + React 18
- TypeScript strict
- pnpm
- Vitest
- ESLint + Prettier
- GitHub Actions
- Vercel

戦闘カーネルはReactから独立した純粋TypeScriptとします。

## 先行判断

- 効果処理順（doc07の10段階）はT016で骨格を確定する。後からの順序変更は決定論テストを全面破壊する
- RepositoryインターフェースはT032で定義し、Phase 5～7はメモリアダプタで進める。Phase 8を差し替え作業に限定するため
- T043（コンテンツ検証）はPhase 6着手時点へ前倒し可。研究網・ステージのデータ制作前に検証器があると手戻りが減る
- シミュレーション指標収集はT046を待たず、ヘッドレスランナー拡張として薄く先行実装してよい（T027の回帰ベンチと共用）
- リファレンステストのゴールデン更新は、仕様変更PRで差分理由を明記した場合のみ許可する（T024で規約化）

## 運用ルール

- 1実装単位＝ `tasks/` の1ファイル＝1ブランチ＝1PR。着手前に `tasks/STATUS.md` へ登録する（既存規約どおり）
- 単位内のチェックリストは、実装・コミットの内訳としてCodexへそのまま渡す
- 【単独】印の5単位（T016 / T024 / T025 / T028 / T041）は束ねず、小さく実装して重点レビューする
- 同一単位で実装に2回失敗したら、チェックリスト境界で分割して再投入する
- 数値はコード定数へ散在させず、マスターデータまたはバランス設定へ集約する
- 各Phase完了時に本書の該当節を更新し、READMEの「現在の状態」を同期する
